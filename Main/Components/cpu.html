CPU (Processeur)

Le CPU est l’unité qui exécute le programme. Il lit des instructions, traite des données et décide quelles actions envoyer vers les autres blocs (mémoire, entrées/sorties). Dans un microcontrôleur, il travaille en permanence avec la mémoire et les périphériques, souvent en temps réel.

Composant central

Rôle dans le système: pourquoi et comment

Le CPU est là pour transformer un programme en actions concrètes. Il suit généralement un cycle: chercher → décoder → exécuter. Il va chercher une instruction en ROM/Flash, puis il manipule des données en RAM et écrit des commandes vers l’I/O.

Dans un système embarqué, ce rôle est encore plus important: le CPU doit souvent répondre à des événements (capteur, bouton, communication) avec un timing stable. Pour ça, il s’appuie sur l’horloge et sur les interruptions.

Ce qu’il lit
Instructions (ROM/Flash), données (RAM), états (I/O)
Ce qu’il produit
Calculs, décisions, commandes vers périphériques
Ce qui le cadence
Horloge + timers + interruptions

Exemple concret

Un ventilateur doit s’activer si la température dépasse 30°C. Le CPU lit la mesure, compare au seuil, puis active une sortie (par exemple en PWM) pour contrôler la vitesse.

lire température → comparer à 30
si > 30 → activer PWM (ventilo)
sinon → arrêter / réduire PWM

Plus de détails

Le CPU d’un MCU contient typiquement des registres (petites mémoires très rapides), une ALU (unité arithmétique et logique), et une logique de contrôle. Quand il exécute une instruction, il ne “devine” pas: tout est défini par l’architecture du processeur.

Dans un MCU, beaucoup de périphériques sont accessibles via des registres mappés en mémoire (memory-mapped I/O). Ça veut dire que lire un capteur ou configurer un port série revient souvent à lire/écrire à une adresse spéciale.

Le CPU gère aussi les interruptions: au lieu de vérifier en boucle “est-ce qu’un événement arrive?”, un périphérique peut interrompre le CPU dès qu’il se passe quelque chose (ex: réception d’un octet UART, fin de conversion ADC, front sur un bouton). Résultat: réaction rapide et code plus propre.

Enfin, la performance ne dépend pas seulement de la fréquence. La vitesse réelle dépend aussi de: la mémoire (latence), l’organisation des bus internes, et du coût de certaines opérations (multiplication, accès périphériques, etc.). Pour un projet scolaire, retenir l’idée globale suffit: plus l’horloge est élevée, plus le CPU peut faire d’étapes par seconde, mais souvent avec plus de consommation.

TL;DR (à retenir)

Le CPU est le moteur logique du MCU: il exécute le code stocké en ROM/Flash, utilise la RAM pour les données temporaires et pilote l’I/O pour interagir avec le monde réel. Il fonctionne de manière synchronisée grâce à l’horloge et réagit vite grâce aux interruptions.

  • Cycle: chercher → décoder → exécuter.
  • Ne stocke pas durablement: ROM/Flash = code, RAM = travail.
  • Contrôle le reste via bus + registres périphériques.
  • Temps réel: interruptions + timers pour réagir sans retard.
